# 项目架构与上下文
<project_context>
## 1. 模块列表
- common：共用模块（通用工具、配置、数据库连接、ResultUtil等）
- user：用户模块
- chat：聊天模块
- prompt：提示词模块
- tenant：租户模块
- company：企业模块
- agent：智能体模块
- gateway：网关模块（负责路由转发、JWT校验、异步请求日志）

## 2. 模块内部标准结构
每个业务模块（user/chat/prompt/tenant/company/agent）均严格遵循以下分层：
- controller/：控制器层（仅处理 HTTP 入参、出参、状态码）
- entity/：数据库模型实体映射类
- mapper/：数据访问层（仅处理数据访问）
- service/：服务层接口
- service/impl/：服务层接口实现（处理业务流程和事务边界）

## 3. 路由识别规则
URL 格式：`/service/{模块名}/...`
- 示例：`/service/user/getUserData` → user 模块
- 示例：`/service/prompt/getPrompt` → prompt 模块
</project_context>

# 强制开发规范（必须严格遵守）
<development_rules>
## 1. 统一接口返回规范
- 所有接口必须使用 `common` 模块的 `ResultUtil` 进行包装。
- 成功返回：`ResultUtil.success(data)`
- 参数说明：
  - `data`: 实际业务数据
  - `msg`: 提示信息（错误提示或增删改成功时使用）
  - `total`: 记录总数（仅分页查询时设置）
  - `token`: 用户凭证（仅登录/注册接口返回）

## 2. 服务注册与命名
- 所有服务必须向 Nacos 注册。
- 服务名称命名规则：`{模块名称}-service`（如：`user-service`）。

## 3. 认证与身份透传
- Gateway 模块：负责 JWT 校验，解析 `userId` 后放入请求头 `X-User-Id` 透传给下游。**严禁** Gateway 直接调用业务模块内部逻辑。
- 业务模块：**禁止**再次进行 JWT 校验。统一通过 `@RequestHeader("X-User-Id") String userId` 获取当前登录人 ID。

## 4. 命名与字段转换
- 数据库字段：下划线命名（snake_case），如 `user_name`, `create_time`。
- 接口入参（Request）：必须使用驼峰命名（camelCase）。
- 接口出参（Response）：数据库字段自动转换为驼峰命名，统一使用 `ResultUtil.success()` 封装。

## 5. MyBatis XML 规范
- 动态 SQL 必须写在 XML 文件中，严禁在 Java Mapper 接口中使用 `@Select` 等注解写复杂 SQL。
- XML 文件存放路径：`src/main/resources/mapper/`。
- 文件命名规则：`{模块名称首字母大写}Mapper.xml`（例如：user 模块对应 `UserMapper.xml`，chat 模块对应 `ChatMapper.xml`）。

## 6. Java 代码风格
- 类名：PascalCase（如 `UserController`, `UserService`）。
- 文件命名：与类名一致，后缀为 `.java`（注意：本项目为 Java 项目，非 Python）。
- 方法/变量：camelCase。
- 事务控制：涉及多表写操作必须在 Service 层使用 `@Transactional`。
- 异常处理：禁止在 Controller 层吞掉异常，统一由全局异常处理器捕获。
</development_rules>
